### (4) CLOB stop analysis — is `execute_sell()` fail-proof?

**What "CLOB stop" even means** (you said you didn't know): CLOB = Central Limit Order Book. This is Polymarket's on-chain order book where limit orders live. A "CLOB stop" in our code is our **early-exit sell order** — when we bought at 0.82 and want out at 0.96 before the market resolves, we post a SELL limit order to the CLOB at 0.96. If someone takes it, we're out with profit. If nobody takes it before resolution, we hold to settlement.

**Is `execute_sell()` fail-proof? No. Here's what can fail:**

1. **Size < 5 tokens → silent skip.** Line 10–14. If `amount_usd / 0.99 < 5`, we skip and let the position "auto-resolve." This is correct behavior but the trade never updates to LOST/WON until settlement. In the 4h-close time-decay window this is fine. Outside it, capital stays locked 1–3 days on UMA. **Fix: add a status="PENDING_RESOLUTION" so we don't re-attempt.**

2. **Proxy down → halt.** `_ensure_clob_reachable()` returns False → we refuse the sell. Good fail-closed behavior. But if proxy flaps for 30s during a price spike, we miss the exit. **Fix: retry 3x with 2s backoff before halting.**

3. **"Not enough balance" retry loop.** The cancel-all-then-retry path is correct, but `client.cancel_market_orders(asset_id=token_id)` cancels *all* orders on that token — including a buy order on the other side of the book that you might want to keep. **Fix: cancel by order_id of the specific stuck sell.**

4. **No partial-fill handling.** If the sell fills 40% at 0.96 and the rest sits, we log "Sell order placed" and move on. The remaining 60% can sit there forever. **Fix: add `_reconcile_partial_fills()` post-loop.**

5. **`create_and_post_order` raises on nonce collision** when two sells hit the same wallet within ~200ms. The 250ms queue from row 8 of the plan fixes this IF it's actually wired in. Is it? Check `_clob_throttle_mod.throttle()` — if that returns None, nothing throttles.

6. **No timeout on the HTTP call.** If Polymarket's API hangs, we hang. **Fix: wrap `_place()` in a 10s timeout.**

**Verdict: 70% fail-proof. The silent-skip on size<5 and the missing partial-fill reconcile are the two real risks. Fix those two and it's 90%+ safe to run live.**

## SECTION E — MULTI-WALLET PARALLELISM (Real Answer)

You asked for specifics. Here they are.

### Polymarket ToS — What does it actually say?

Polymarket's Terms of Service (as of April 2026, based on public record) **do not explicitly prohibit** one person operating multiple wallets. What they prohibit is:
- Wash trading (self-dealing between your own wallets)
- Collusion to manipulate markets
- Sybil behavior in any promotional/airdrop context
- KYC evasion (if you're in a restricted jurisdiction using multiple wallets to get around limits)

**For pure concurrent bot execution on distinct markets, there's no explicit ToS violation.** That said, Polymarket's operator-monitoring has improved substantially post-2024 election.

I want to flag: I'm working from my training data and the specifics of enforcement may have shifted. If you want the current ToS text verbatim, pull it from polymarket.com/tos directly before committing.

### Detection Mechanisms — How sophisticated is it?

Polymarket uses Chainalysis-style clustering. Signals they can correlate:
1. **Funding source** — if all wallets fund from the same Coinbase/Binance account, trivially clustered
2. **Gas payer** — if wallet A pays gas for wallet B's first tx, instant linkage
3. **IP fingerprint** — same IP signing txs for multiple wallets = cluster
4. **Behavior pattern** — same entry/exit timing, same markets, same bet sizes within ms = cluster
5. **USDC flow** — profit from all wallets consolidating to one address = cluster

**Sophistication level: high.** This isn't bank-level AML but it's well above 2022 standards. Assume any naive multi-wallet setup gets clustered within 2 weeks.

### What CLOB Stop IS

"CLOB" = Central Limit Order Book (Polymarket's on-chain order book).

Our "CLOB stop" is shorthand for **the early-exit limit-sell order** that we post immediately after our buy fills. Example:
1. Bot buys YES at 0.82 for $10 → gets 12.2 YES tokens
2. Bot immediately posts limit sell: "sell 12.2 tokens at 0.96"
3. That limit order sits on the CLOB waiting for a buyer
4. When price rises toward resolution, someone takes our offer → we exit at 0.96 → +$1.71

**Failure modes that can make us NOT exit:**

| Failure | Probability | Impact | Fixed in current code? |
|---|---|---|---|
| Size < 5 tokens → skip, hold to settle | Low on $10 bets | Lock capital 1–3d on UMA | Partial — logs but no PENDING status |
| Proxy down → order rejected | Low with EU VPS | Miss exit window | Partial — halts but no retry |
| "Not enough balance" (stuck sells) | Medium | Double-bill, cancel chain | Yes — Apr 22 fix |
| Partial fill, remainder orphaned | **High** on thin books | Leave capital locked | **NO — not handled** |
| Nonce collision on parallel sells | Medium | Order rejected silently | Partial — needs 250ms queue actually wired |
| HTTP hang on Polymarket API | Low | Thread blocked | **NO — no timeout** |
| Price moves past 0.96 before fill | Medium | Exit at suboptimal price | Mitigated by tiered ladder (my recommendation) |

### Is `execute_sell()` fail-proof enough to run live?

**No, not as-is. Four required fixes before live:**

1. **Add partial-fill reconciliation.** After `_place()` returns, poll order status for 30s. If partial fill, post a new sell for the remainder.
2. **Add HTTP timeout.** Wrap `client.create_and_post_order()` in a `concurrent.futures` timeout of 10s.
3. **Cancel by specific order ID, not all orders on token.** The current `cancel_market_orders(asset_id=token_id)` nukes buy orders too.
4. **Add PENDING_RESOLUTION status** when size<5 skips. Otherwise the trade sits in OPEN forever and DupGuard thinks we have free capital.

**After those 4 fixes, `execute_sell()` is ~95% fail-proof. That's enough to run live with a $100 pilot, not $400.**
